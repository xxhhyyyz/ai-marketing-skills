AI Hackathon / 编程马拉松 / AI 创意大赛的从 0 到 1 筹备和执行。

## 工作流程总览

```
赛事定位 → 赛题设计 → 赛程规划 → 技术支持 → 推广招募 →
赛事执行 → 评审颁奖 → 复盘孵化
```

## Step 1：赛事定位

### 1.1 赛事类型

| 类型 | 时长 | 规模 | 特点 | 适合阶段 |
|------|------|------|------|---------|
| **闪电 Hack** | 4-8 小时 | 20-50 人 | 快速原型、低门槛 | 冷启动/社区 |
| **周末 Hackathon** | 24-48 小时 | 50-200 人 | 经典形式、深度开发 | 增长期 |
| **线上马拉松** | 1-4 周 | 不限 | 远程参与、规模大 | 全球扩展 |
| **命题赛** | 2-4 周 | 50-500 人 | 聚焦特定场景 | 商业合作 |
| **创意大赛** | 2-8 周 | 不限 | 重创意轻代码 | 泛用户 |

### 1.2 定位决策

```markdown
## 赛事定位文档

### 核心目标（选 1-2 个）
- [ ] 开发者获取和激活
- [ ] 产品/API 的真实使用场景验证
- [ ] 开源生态和社区建设
- [ ] 品牌传播和行业影响力
- [ ] 商业场景解决方案征集
- [ ] 人才发现和招聘

### 目标参赛者
- **主要群体**：学生 / 独立开发者 / 企业开发者 / 非技术创作者
- **技术水平**：初学者友好 / 中级 / 高级
- **地理范围**：本地 / 全国 / 全球

### 与产品的关系
- **必须使用我们的产品/API**：是 / 否 / 鼓励使用
- **开放第三方工具**：完全开放 / 限定范围
```

## Step 2：赛题设计

### 2.1 赛题模式

| 模式 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **开放式** | 自由选题，用 AI 解决任何问题 | 创意多元 | 作品难对比 |
| **命题式** | 指定具体问题 | 聚焦、可评 | 限制创意 |
| **赛道式** | 设 3-5 个赛道，每赛道有方向 | 平衡自由度和聚焦 | 设计复杂 |
| **场景式** | 提供真实业务场景和数据 | 实用性强 | 准备工作多 |

### 2.2 赛道设计模板（推荐赛道式）

```markdown
## 赛道设计

### 赛道 1：[名称] — [一句话描述]
- **方向**：
- **示例场景**：
  - 场景 A：...
  - 场景 B：...
- **可用资源**：API / 数据集 / 模型
- **评分侧重**：创新性 40% / 完成度 30% / 实用性 30%

### 赛道 2：...

### 赛道 3：...

### 自由赛道（可选）
- 不限方向，鼓励探索
- 需要使用至少一个指定 API/工具
```

### 2.3 好赛题的标准

```
✅ 好赛题：
- 24-48 小时内可以做出可演示的原型
- 有清晰的评判标准
- 能体现 AI 产品的核心能力
- 参赛者可以自由发挥但有方向

❌ 差赛题：
- 太大（"用 AI 解决医疗问题"）
- 太小（"调 API 返回结果"）
- 太模糊（"做一个 AI 应用"）
- 门槛太高（需要大量前置知识或数据）
```

## Step 3：赛程设计

### 3.1 周末 Hackathon 标准赛程

```
📅 赛前（T-14 ~ T-1）
├── T-14  报名开启 + 推广启动
├── T-7   赛前培训 / Workshop（线上）
│         · API 使用教程
│         · 历届优秀作品分享
│         · 组队配对
├── T-3   参赛者微信群/Discord 建立
└── T-1   赛前提醒 + 技术环境检查

📅 Day 1（比赛日）
├── 09:00  开幕式
│          · 主办方介绍
│          · 赛题解读
│          · 评审标准说明
│          · 技术支持介绍
├── 09:30  开发开始
├── 12:00  午餐 + 自由交流
├── 14:00  中期 Check-in（可选）
│          · 各队进度分享（1 分钟/队）
│          · Mentor 巡场答疑
├── 18:00  晚餐
├── 20:00  技术答疑时间
└── 22:00  Day 1 结束（或通宵）

📅 Day 2（提交日）
├── 09:00  继续开发
├── 12:00  午餐
├── 14:00  ⚠️ 提交截止
├── 14:30  Demo 排练（各队测试演示环境）
├── 15:00  Demo Show
│          · 每队 3-5 分钟演示 + 2 分钟 Q&A
│          · 评审打分
├── 17:00  评审合议
├── 17:30  颁奖典礼
│          · 各奖项公布
│          · 优秀作品点评
│          · 合影
└── 18:00  结束 + 自由交流
```

### 3.2 线上马拉松赛程

```
📅 Week 1：报名 + 组队
├── 报名页面上线
├── 线上说明会
└── 组队配对平台

📅 Week 2：开发期
├── 开幕直播
├── 每日 Mentor Office Hour
├── 技术支持群实时答疑
└── 中期分享（可选）

📅 Week 3：提交 + 评审
├── 作品提交截止
├── 初审（线上评审）
├── 入围通知
└── 决赛准备

📅 Week 4：决赛 + 颁奖
├── 线上/线下决赛 Demo
├── 评审打分
├── 颁奖直播
└── 作品展示页面上线
```

## Step 4：技术支持方案

### 4.1 API/资源准备

```markdown
### 技术支持清单
- [ ] 参赛者专属 API Key 批量生成
- [ ] API 额度设置（建议比预估用量多 3 倍）
- [ ] 开发文档专题页面
- [ ] Quick Start 教程（5 分钟跑通第一个 Demo）

- [ ] Sandbox 环境（如有）
```

### 4.2 Mentor 安排

| 角色 | 职责 | 人数建议 |
|------|------|---------|
| **技术 Mentor** | API 使用答疑、Debug 协助 | 每 5 队 1 人 |
| **产品 Mentor** | 产品方向建议、场景优化 | 每 10 队 1 人 |
| **商业 Mentor** | 商业化建议、Pitch 指导 | 2-3 人（决赛） |

### 4.3 技术保障

```markdown
### 比赛期间技术保障
- [ ] API 监控面板（实时关注调用量和错误率）
- [ ] 紧急联系人 on-call（全程在线）
- [ ] 常见问题 FAQ 文档（持续更新）
- [ ] 备用 API 方案（如果主 API 过载）
- [ ] 技术支持群（即时响应）
```

## Step 6：评审标准

### 6.1 通用评分维度

| 维度 | 权重 | 评分要点 |
|------|------|---------|
| **创新性** | 25% | 方案是否有新意，是否解决了真实问题 |
| **完成度** | 25% | 是否有可运行的 Demo，功能是否完整 |
| **技术实现** | 20% | AI 技术使用是否合理和深入 |
| **实用性** | 15% | 是否有真实使用场景和商业潜力 |
| **展示效果** | 15% | Demo 演示和 Pitch 是否清晰有说服力 |

### 6.2 评分表模板

```markdown
## 评审打分表

**作品名称**：
**团队名称**：
**赛道**：

| 维度 | 满分 | 得分 | 评语 |
|------|------|------|------|
| 创新性 | 25 | | |
| 完成度 | 25 | | |
| 技术实现 | 20 | | |
| 实用性 | 15 | | |
| 展示效果 | 15 | | |
| **总分** | **100** | | |

**综合评语**：

**评审签名**：
```

### 6.3 评审流程

```
初审（线上）
├── 所有提交作品按评分维度打分
├── 每个作品至少 2 位评审
├── 取平均分排序
└── 前 N 名进入决赛

决赛（Demo Show）
├── 每队 3-5 分钟 Demo + 2 分钟 Q&A
├── 评审现场打分
├── 评审合议讨论争议作品
└── 最终排名确定
```

### 6.4 评审团组建

```markdown
### 评审构成建议（5-7 人）
- 技术专家 × 2：评估技术实现和 AI 使用深度
- 产品/设计专家 × 1：评估用户体验和产品化程度
- 商业/投资人 × 1：评估商业价值和市场潜力
- 行业专家 × 1：评估行业适用性和创新度
- 主办方代表 × 1-2：整体把控和品牌对齐
```

## 野路子

- **赛前让参赛者在群里自由组队，别帮他们分配**。自然社交形成的团队默契度远高于你按技能互补配的。你只需要建一个「找队友」频道
- **准备一个「速成 Demo 模板」**。一个好看的 Next.js/Streamlit 模板，让完全不会前端的后端工程师也能做出能演示的东西。这个模板本身就是你产品的最佳实践
- **获奖作品别只发奖金**。帮他们写一篇技术博客，帮他们推上 Product Hunt，帮他们在 Twitter 上发 thread。这些后续传播的价值比奖金大 100 倍
- **比赛期间最好的内容是参赛者的「开发日志」**。鼓励他们边做边发推/朋友圈，甚至设一个「最佳记录奖」。比赛还没结束，传播已经开始了
- **评审别只请大佬**。请一个你的目标用户——他的视角比任何专家都真实。「这东西我会用吗？」比「技术架构是否优雅」重要得多
- **赛后 72 小时内发第一篇内容**。参赛者的朋友圈热度只有 3 天，过了就凉了
- **用飞书多维表格追踪所有参赛团队的后续动态**。3 个月后看看谁还在用你的 API、谁把作品做成了产品——这些才是 Hackathon 的真正 ROI
- **如果预算有限，别在场地和餐饮上省，在奖金上省**。好的体验 > 高额奖金。参赛者记住的是那天的氛围和认识的人，不是奖金数额

## Notes

- Hackathon 的核心价值是**社区和生态**，而非单纯的竞赛
- 技术支持是成败关键：API 挂了 = 比赛废了
- 比赛的传播价值在于**参赛者的作品和故事**，而非赛事本身
- 控制规模和质量：50 队高质量 > 200 队水作品
- 赛后跟进决定长期价值：获奖作品如果能变成真实产品，这才是最好的 ROI
- 首次举办建议从「闪电 Hack」开始，积累经验后再做大型赛事
