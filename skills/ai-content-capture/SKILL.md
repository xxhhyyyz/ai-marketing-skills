# AI Content Capture — AI 营销内容抓取与提炼

从多种来源自动抓取和提炼 AI 产品的核心营销素材，输出结构化内容库。

**前置依赖**：先读取 [ai-marketing-context](../ai-marketing-context/SKILL.md) 了解产品背景。

**相关 Skills**：
- [ai-content-distribution](../ai-content-distribution/SKILL.md) — 素材产出后的传播策略
- [ai-event-sop](../ai-event-sop/SKILL.md) — 活动中的内容素材准备
- [ai-hackathon](../ai-hackathon/SKILL.md) — 赛事的技术亮点提炼

---

## 工作流程

```
识别内容来源 → 抓取原始信息 → 提炼营销角度 → 结构化输出 → 归档素材库
```

---

## Step 1：识别内容来源

根据用户输入判断来源类型：

| 来源类型 | 触发信号 | 处理方式 |
|---------|---------|---------|
| Changelog / Release Notes | 用户提供更新日志或链接 | 逐条提炼卖点 |
| 技术文档 / 论文 | 用户提供技术文档 | 转译为用户可感知的价值 |
| 竞品动态 | 用户提到竞品名或竞品链接 | 提取差异化机会 |
| 用户反馈 | 用户提供评价/社区讨论 | 挖掘真实场景和证言 |
| Benchmark / 评测 | 用户提供评测数据 | 提取可引用数据点 |
| 行业报告 | 用户提供报告链接或文件 | 提取趋势和数据支撑 |
| 自由输入 | 用户口述或粘贴文本 | 智能分类后处理 |

---

## Step 2：针对不同来源的提炼方法

### 2.1 Changelog / Release Notes → 营销卖点

**输入**：产品更新日志（如 GitHub release、内部 changelog）

**提炼框架**：

对每个更新条目，输出：

```markdown
### [功能名称]

- **一句话卖点**：（面向用户的价值，非技术描述）
- **目标用户**：谁最受益
- **使用场景**：具体在什么情境下解决什么问题
- **Before/After**：使用前 vs 使用后的对比
- **数据支撑**：（如有：性能提升 X%、成本降低 X%）
- **营销角度建议**：
  - 社交媒体 hook：
  - 博客标题备选：
  - Demo 演示点：
```

**提炼原则**：
- 技术特性 → 用户价值：「支持 function calling」→「让 AI 自动帮你完成复杂任务」
- 性能数据 → 体感对比：「推理速度提升 40%」→「回复速度从 3 秒缩短到 1.8 秒」
- 架构改进 → 商业价值：「支持多模态输入」→「一个 API 搞定文字、图片、语音」

### 2.2 技术文档 / 论文 → 用户价值

**输入**：技术博客、白皮书、学术论文

**提炼框架**：

```markdown
### [论文/文档标题]

- **核心创新**：一句话说清楚技术突破
- **用户能感知到什么**：翻译成非技术人员能理解的影响
- **可引用的关键数据**：
- **适合的传播形式**：技术博客 / 信息图 / 短视频 / Twitter thread
- **目标受众**：开发者 / 产品经理 / 决策者 / 媒体
- **类比/比喻建议**：（帮助非技术受众理解的类比）
```

### 2.3 竞品动态 → 差异化素材

**输入**：竞品发布公告、新闻报道、产品更新

**提炼框架**：

```markdown
### 竞品动态：[竞品名] — [事件]

- **发生了什么**：一句话概述
- **对我们的影响**：正面机会 / 需要防御 / 中性
- **差异化机会**：
  - 我们已有的优势：
  - 可以强化的叙事：
  - 建议的响应策略：主动出击 / 静观其变 / 借势传播
- **内容创作建议**：
  - 可输出的内容类型：
  - 建议时效性：立即 / 1 周内 / 长期
```

> ⚠️ **注意**：参考 ai-marketing-context 中的「竞品提及策略」，
> 遵循品牌对竞品提及的态度。

### 2.4 用户反馈 → 场景与证言

**输入**：用户评价、社区讨论、支持工单、社交媒体提及

**提炼框架**：

```markdown
### 用户声音：[来源平台]

- **原文摘录**：「...」
- **用户角色**：开发者 / 企业用户 / 创作者
- **使用场景**：
- **核心价值点**：用户因为什么而满意/推荐
- **可用于**：
  - [ ] 官网 testimonial
  - [ ] 社交媒体引用
  - [ ] 案例故事素材
  - [ ] 销售 deck 引用
- **跟进建议**：是否值得深挖为完整案例
```

### 2.5 Benchmark / 评测 → 数据弹药

**输入**：性能评测、排行榜、对比测试

**提炼框架**：

```markdown
### 评测数据：[评测名称/来源]

- **核心结论**：一句话
- **关键数据点**：
  | 指标 | 我们 | 竞品 A | 竞品 B |
  |------|------|--------|--------|
  |      |      |        |        |
- **数据可视化建议**：柱状图 / 雷达图 / 对比表格
- **引用注意事项**：数据时效性、评测条件说明
- **适合的传播场景**：官网 / 技术博客 / 社交媒体 / 销售材料
```

---

## Step 3：结构化输出

将提炼结果汇总为素材卡片格式：

```markdown
# 内容素材库 — [日期]

## 素材概览
- 总计：X 条素材
- 来源分布：Changelog X 条 / 竞品 X 条 / 用户反馈 X 条
- 优先级：🔴 高 X 条 / 🟡 中 X 条 / 🟢 低 X 条

## 高优先级素材

### 素材 #1：[标题]
- **来源**：
- **类型**：功能卖点 / 数据支撑 / 用户证言 / 差异化 / 趋势借势
- **一句话**：
- **完整描述**：
- **建议用途**：社交媒体 / 博客 / 官网 / 销售 / 活动
- **时效性**：长期有效 / 1 个月 / 1 周 / 即时
- **优先级**：🔴 高

...
```

---

## Step 4：素材归档

建议将素材库保存到项目中：

```
.agents/content-library/
├── YYYY-MM-DD-changelog.md
├── YYYY-MM-DD-competitor.md
├── YYYY-MM-DD-user-feedback.md
└── index.md  (素材索引和统计)
```

## 快速命令

| 用户说 | 动作 |
|-------|------|
| "帮我从这个 changelog 提炼卖点" | 执行 2.1 Changelog 流程 |
| "分析一下这个竞品动态" | 执行 2.3 竞品流程 |
| "这些用户评价里有什么可用的" | 执行 2.4 用户反馈流程 |
| "把这个技术文档翻译成营销语言" | 执行 2.2 技术文档流程 |
| "整理一下最近的营销素材" | 读取素材库，生成摘要 |

---

## Notes

- 始终从营销角度而非技术角度提炼，核心问题是「用户为什么应该在乎」
- 保持素材的事实准确性，不夸大不捏造数据
- 每条素材标注时效性，过期素材及时清理
- 提炼时参考 ai-marketing-context 中的品牌调性，保持一致的叙事风格
